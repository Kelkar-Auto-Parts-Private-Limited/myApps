1. Product Overview

KAP myApps is an enterprise web platform built for Kelkar Auto Parts Private Limited (KAP). It provides a unified suite of internal business applications covering vehicle logistics, warehouse management, human resources, and physical security operations.

Architecture

  • Frontend: Static HTML/CSS/JS site hosted on GitHub Pages (repo: Kelkar-Auto-Parts-Private-Limited/myApps).
  • Backend: Supabase (PostgreSQL) for authentication, database, realtime subscriptions, and file storage.
  • Offline-first: All data is cached in localStorage. The app loads instantly from cache and syncs with Supabase in the background.
  • Auth: Custom username/password authentication via Supabase RPC (verify_session, do_login). Session tokens stored in localStorage. CAPTCHA and account lockout on repeated failures.
  • Realtime sync: Supabase Realtime (postgres_changes) keeps all connected devices in sync. Changes on one device appear on others within seconds.

2. App Descriptions

🏠 myApps Portal

Purpose: Central entry point and administration hub. After login, users see the portal with app tiles for the modules they are authorized to access. Admins manage all user accounts from here.

Key Features

  • App Launcher -- Grid of app tiles filtered by the user's apps[] array. Each tile links to the corresponding module HTML page.
  • User Management (Admin only) -- Create, edit, deactivate users. Assign roles, HWMS roles, HRMS roles, plant location, and app access. Upload user photos. Import/export users.
  • Profile & Password -- Any user can update their own full name, mobile, email, photo, and password.
  • DB Storage Analysis (Super Admin only) -- Inspect localStorage usage and table-level record counts.

Target Users: All platform users (login), Super Admins and Admins (user management).

5. Integration Points

5.1 Supabase

ServiceUsage
Database (PostgreSQL) All application data stored across 30+ tables. Uses upsert with onConflict:'code' for idempotent writes. RPC functions for auth (do_login, verify_session).
Realtime Subscribes to postgres_changes (INSERT, UPDATE, DELETE) on hot tables (trips, segments, spotTrips). Enables cross-device live sync. Auto-reconnects on channel errors with 10-second retry.
Storage Used for photo uploads (driver photos, guard photos, user avatars, challan/vehicle photos, part photos, packing photos, carrier invoice photos). Photos are stored as base64 data URIs in the database columns directly (not Supabase Storage buckets).
Auth Custom auth (not Supabase Auth). Session token generated by do_login RPC, verified by verify_session RPC. persistSession: false on the Supabase client.

5.2 GitHub Pages

The entire frontend is deployed as a static site on GitHub Pages from the repository Kelkar-Auto-Parts-Private-Limited/myApps, branch main. No server-side rendering or build step -- plain HTML, CSS, and JS files served directly.

5.3 Offline-First / localStorage Cache

MechanismDetails
Session persistence kap_session stores username + token. kap_current_user caches the full user object for instant session restore without waiting for Supabase CDN.
Data cache Each table is cached in localStorage (key per table). On boot, the app loads from cache first (instant UI), then syncs from Supabase in the background (_bgSyncFromSupabase).
Connectivity monitoring A 30-second ping interval checks Supabase connectivity. Status widget in every module sidebar shows connected/error/offline state. After 3 consecutive failures, the client attempts re-initialization.
Force sync _forceSyncAll() pushes all in-memory data to Supabase (recovery tool). Accessible via the VMS sidebar connection widget.
App-specific lazy loading Each module sets _APP_TABLES to load only its own tables, reducing boot time. Falls back to all tables if not set.

5.4 Security Features

  • Math CAPTCHA on the login page to prevent automated access.
  • Account lockout after repeated failed login attempts (60-second cooldown).
  • Password policy enforced: minimum 6 characters, maximum 12.
  • Session tokens verified on every page navigation via RPC.
  • Role-based access control enforced both in the UI (nav items, buttons hidden) and in business logic (permission checks before save/delete).
  • Deletion guards: referential integrity checks prevent deleting users or masters that are referenced by other records. The last Super Admin cannot be deleted.
