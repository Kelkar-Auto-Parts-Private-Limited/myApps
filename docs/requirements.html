Requirements

Project: KAP myApps Portal  |  Last updated: 2026-04-03  |  Version: 1.0

1. Portal (index.html)

The portal is the single entry point for all KAP staff. It handles authentication and provides navigation to each app module.

1.1 Authentication

  • Users log in with a username and password stored in the vms_users Supabase table.
  • A math CAPTCHA must be solved on every login to prevent scripted access.
  • "Remember me" persists the session in localStorage so users do not need to log in again on the same device.
  • Failed login attempts show an error message; the CAPTCHA refreshes.

1.2 App launcher

  • After login, the portal displays tiles for each available app.
  • Apps the user does not have access to are shown as disabled (greyed out).
  • Clicking an app tile opens that app's HTML file.

1.3 User management

  • Admin users can create, edit, and deactivate user accounts from within the portal.
  • Each user has a role that determines which apps and features they can access.

2. VMS — Vehicle Management System (VMS.html)

Manages all vehicle movements, driver assignments, and trip billing for KAP's transport operations.

2.1 Core entities

  • Vehicles — registered vehicles with type, number plate, and vendor assignment.
  • Drivers — driver profiles linked to vehicles and vendors.
  • Vendors — transport companies that supply vehicles and drivers.
  • Vehicle types — categories of vehicles (e.g. truck, tempo, car).
  • Locations — pickup and drop-off points used in trip planning.

2.2 Trip management

  • Regular trips are logged with origin, destination, vehicle, driver, date, and distance/rate.
  • Spot trips (ad-hoc, unscheduled) can be logged separately.
  • Trip rates are configurable per route and vehicle type.
  • Trips have segments (multi-leg journeys can be broken into segments).

3. HWMS — Hardware/Warehouse Management System (HWMS.html)

Manages import shipments, warehouse stock, invoicing, and payments for hardware parts.

3.1 Parts and stock

  • Parts catalogue with HSN codes, units of measure (UOM), and packing details.
  • Steel rates maintained separately for rate calculations.
  • Parts are tracked through warehouse statuses: on-water, in-warehouse, sold.

3.2 Import shipments

  • Containers are tracked from port of loading through to warehouse receipt.
  • Ports of loading and discharge are configurable.
  • Carriers (shipping lines) are maintained in a reference table.

3.3 Invoicing and payments

  • Sales invoices are raised against customers.
  • Sub-invoices can be attached to a main invoice.
  • Payment receipts are recorded and matched against invoices.
  • Invoice statuses: draft, GRN pending, payment pending, partially paid, fully paid.

3.4 Material requests

  • Internal material requests (MR) can be raised and tracked through fulfilment.

3.5 Reference data

  • Customers, company details, HSN codes, UOM, packing types are all maintained as configurable reference tables.

4. Security Surveillance (Security.html)

Manages security guard rounds, checkpoints, and schedules at KAP premises.

  • Checkpoints — physical locations on the premises that guards must visit during a round.
  • Guards — security staff profiles and shift assignments.
  • Round schedules — planned patrol routes and timing for guards to follow.
  • Guards log when each checkpoint is visited; supervisors can review compliance.

5. Cross-cutting requirements

  • All apps must work on desktop and mobile browsers (responsive design).
  • All apps share a single Supabase backend — data is live and multi-user.
  • Realtime updates are supported via Supabase realtime subscriptions.
  • Export to Excel is available in list views (via the shared export utility in Common.js).
  • All apps use the same visual design language (teal accent colour, shared CSS variables).
