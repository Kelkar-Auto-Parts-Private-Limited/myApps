1 System Overview

KAP myApps is a multi-module enterprise platform serving Kelkar Auto Parts' operations across vehicle management, warehouse logistics, HR, and security surveillance. It is built as a static HTML/JS/CSS application hosted on GitHub Pages with no build step, no bundler, and no server-side rendering.

Key Architectural Principles

  • Supabase-first writes: All data mutations go directly to Supabase. In-memory DB is updated only after server confirmation.
  • In-memory DB for reads: The global DB object holds all active data in memory. Views render from DB, never from direct Supabase queries.
  • No localStorage for DB tables: Data lives in Supabase only. localStorage is used only for session tokens (kap_session_user, kap_session_token), remember-me credentials (kap_rm_user, kap_rm_token), and a transient navigation cache (kap_db_cache, max 60s TTL).
  • App-specific table filtering: Each module sets _APP_TABLES at boot to load only its own tables, reducing bandwidth and boot time.
  • Three-layer JS architecture: common.js (shared data layer) → *-logic.js (business rules) → *-ui.js (DOM rendering and event handling).

Module Hooks

common.js defines four hook functions that each UI module overrides to receive lifecycle events:

  • _onPostBoot() — Called after bootDB() completes. Modules use this to render initial views.
  • _onRefreshViews() — Called after background sync or realtime events. Modules re-render active pages.
  • _onCurrentUserUpdated() — Called when the current user record is updated via realtime.
  • _kapPopupOpen — Boolean flag. When true, background refreshes are suppressed to avoid disrupting modal interactions.

4 Database Schema

All tables live in the Supabase public schema. The JS key in SB_TABLES maps to the actual Postgres table name. Every table uses code (TEXT) as the application-level primary key, with an auto-increment id (BIGINT) as the Supabase-internal PK. All tables have an updated_at column used for incremental sync.

Row Mapping: _toRow(tbl, record) converts camelCase JS objects to snake_case Postgres rows. _fromRow(tbl, row) does the reverse. Both functions handle all 28 tables.

5 Authentication Flow

Authentication is handled by custom PostgreSQL RPC functions using the pgcrypto extension for bcrypt hashing. Passwords never leave the database in plaintext — all comparison happens server-side via crypt().

Auth Functions (Supabase RPC)

RPC FunctionJS CallerPurpose
create_session(p_username, p_password)_authLogin()Verifies bcrypt password, generates random 32-byte hex token, stores in session_token column, returns token.
verify_session(p_username, p_token)_authVerifySession()Returns user row if username + token match. Used for session restore on page load.
change_password(p_username, p_old, p_new)_authChangePassword()Verifies old password server-side, hashes and stores new password with gen_salt('bf', 10).
admin_reset_password(p_admin_code, p_target_code)_authResetPassword()Admin-only. Verifies caller has Super Admin/Admin role, resets target to default password (Kappl@123), clears session_token.
set_hashed_password(p_user_code, p_plain)_authSetPassword()Sets bcrypt hash directly. Used when creating new users.

Password Policy

  • 6-12 characters
  • At least one uppercase letter (A-Z)
  • At least one lowercase letter (a-z)
  • At least one number (0-9)
  • At least one special character
  • Default password: Kappl@123 — users with this password are forced to change it on first login
  • Admin reset restores the default password and clears the session token (forces re-login)

6 Sync Architecture

The sync system uses a multi-tier strategy to balance real-time responsiveness with resource efficiency. There are four sync mechanisms running concurrently.

Sync Mechanisms

MechanismIntervalTablesFunctionPurpose
Boot Sync Once at startup All active tables bootDB() Initial data load. Parallel fetch of all tables with 8s timeout. Falls back to empty DB if Supabase unavailable.
Realtime Instant (WebSocket) Hot tables only (trips, segments, spotTrips) _sbStartRealtime() Postgres Changes via Supabase channel. Subscribes to INSERT/UPDATE/DELETE on hot tables only (3 tables = 9 listeners). Auto-reconnects on failure.
Hot Poll Every 60s Hot tables (trips, segments, spotTrips) _bgSyncHot() Fallback for missed realtime events. Fetches hot tables and always refreshes views (no stale-detection).
Full Sync Every 10 min (10th hot poll tick) All active tables _bgSyncFromSupabase() Refreshes all tables including masters (users, vehicles, locations, etc.) that change infrequently.
Connectivity Ping Every 30s users (1 row) _sbStartPing() Lightweight health check. After 3 consecutive failures, sets status to "offline" and attempts Supabase client re-initialization.

Boot Sequence Detail

    │
    ├── Step 1: Check localStorage for kap_db_cache (navigation handoff)
    │     ├── Found + age < 60s → Load from cache, boot instantly
    │     │     └── Schedule background Supabase sync in 3s
    │     └── Not found or stale → Continue to Step 2

    ├── Step 2: Initialize Supabase client
    │     ├── Try _initSupabase() (window.supabase)

    │     ├── Each table: SELECT [sel] FROM [table] LIMIT 10000

    │     └── Race against 8s timeout
    │           ├── All done within 8s → Full load complete
    │           └── Timeout → Use whatever loaded, retry rest in 1s

Visibility API Optimization

Background polling is paused when the browser tab is hidden to conserve bandwidth and avoid unnecessary Supabase API calls:

Tab becomes hidden (document.hidden = true)
    ├── clearInterval(bgPollTimer)     — stops 60s hot poll
    ├── clearInterval(sbPingTimer)     — stops 30s connectivity ping
    └── Record timestamp (_visHiddenAt)

Tab becomes visible again
    ├── Restart bgPoll timer
    ├── Restart ping timer
    └── Check how long tab was hidden:
          ├── Hidden > 30s → Full sync (_bgSyncFromSupabase)
          └── Hidden <= 30s → Hot sync only (_bgSyncHot)

Realtime Event Handling

The _rtApply() function processes realtime events from Supabase:

  • INSERT / UPDATE: Convert row via _fromRow(), upsert into DB[tbl] by matching id. If the updated record is the current user, also updates CU and refreshes avatar/name displays.
  • DELETE: Remove from DB[tbl] by code (preferred) or _dbId (fallback when REPLICA IDENTITY is not FULL).
  • View refresh: Debounced at 200ms per table. Only re-renders the currently active page. Skipped entirely when a modal is open.
Why only hot tables for realtime? Subscribing to all 28 tables creates 84 listeners on a single Supabase channel, which causes event throttling and 10-15s delivery delays. By subscribing only to trips, segments, and spotTrips (the tables that change frequently in real-time operations), the system stays responsive. Master data (users, vehicles, locations) is kept fresh by the 10-minute full sync cycle.

Navigation Cache

When navigating between pages via _navigateTo(), the current in-memory DB is serialized to localStorage as kap_db_cache. The destination page reads this cache during bootDB() and boots instantly (without a Supabase round-trip), then syncs fresh data in the background after 3 seconds. The cache has a 60-second TTL and is deleted after use.

Connection Recovery

The system handles connectivity loss gracefully:

  • Supabase CDN fails to load: A fallback CDN (unpkg.com) is loaded dynamically. If both fail, the app starts in offline mode with an empty database.
  • Network interruption: The 30s ping detects failures after 3 consecutive misses. Sets status to "offline", attempts Supabase client re-initialization, and triggers a background sync on recovery.
  • Write failures: Each _dbSave() / _dbDel() retries once automatically. On final failure, the in-memory DB is NOT updated (preventing divergence from the server), and an error toast is shown.
  • Realtime channel failure: Channel errors trigger automatic reconnection after 10 seconds. The hot poll continues to function as a fallback during the reconnection period.
